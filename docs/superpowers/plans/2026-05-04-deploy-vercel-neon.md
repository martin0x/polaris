# Initial Production Deployment (Vercel + Neon) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Operational tasks (those that touch external dashboards: Neon, Vercel, Google Cloud) should be performed by the human operator — agents cannot click web UIs. Each operational task lists exact field values for the human to enter.

**Goal:** Get Polaris running at a publicly reachable URL, sign-in-able by Raymart's Google account, with a Singapore-region Postgres backing it, and the existing nightly cron firing on schedule.

**Architecture:** Vercel Hobby (region `sin1`) hosts the Next.js app. Neon Free (`aws-ap-southeast-1`) hosts Postgres, with two URLs: a pooled URL for app runtime queries, and a direct URL for Prisma migrations. NextAuth + Google OAuth gates access to a single email via `ALLOWED_EMAIL`. Vercel Cron triggers `/api/cron/compute-active-topics` nightly using the existing `vercel.json`. Object storage and backups are intentionally out of scope (separate follow-up plans).

**Tech Stack:** Next.js 16.2.4, Prisma 7.8 + `@prisma/adapter-pg`, NextAuth v5 beta, Vercel Cron, Neon Postgres 17, Google OAuth.

## Task 1: Split runtime and migration DB URLs in prisma.config.ts

**Files:**
- Modify: `prisma.config.ts`
- Modify: `.env.example`

**Why:** Neon's pooled connection string uses PgBouncer in transaction mode, which doesn't support the prepared statements that `prisma migrate` issues. The fix is to use a **direct** (non-pooled) URL for migrations and the **pooled** URL for the app at runtime. Locally, both can be the same Postgres URL, so the change is backwards-compatible for `bun run dev`.

- [ ] **Step 1: Modify `prisma.config.ts`**

Replace the entire file contents:

```ts
// This file was generated by Prisma, and assumes you have installed the following:
// bun add -D prisma dotenv
import "dotenv/config";
import { defineConfig } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
  },
  datasource: {
    url: process.env["DIRECT_DATABASE_URL"] ?? process.env["DATABASE_URL"],
  },
});
```

- [ ] **Step 2: Modify `.env.example` to document the new var**

Find the `# Database` section. Replace it with:

```
# Database
# DATABASE_URL is the runtime URL the app uses. In production this is the Neon
# pooled URL (with ?pgbouncer=true&connection_limit=1). Locally it's the
# Postgres in docker-compose.
DATABASE_URL=postgresql://polaris:polaris@localhost:5440/polaris
# DIRECT_DATABASE_URL is used by `prisma migrate` (and `prisma db push`).
# Required in production because pooled connections don't support prepared
# statements. Locally, leave unset — migrations will fall back to DATABASE_URL.
# DIRECT_DATABASE_URL=
# Optional — separate database for integration tests. If unset, integration tests skip.
DATABASE_URL_TEST=postgresql://polaris:polaris@localhost:5440/polaris_test
```

- [ ] **Step 3: Verify locally that migrations still work**

Run:
```bash
bun run prisma migrate status
```
Expected: `Database schema is up to date!` (or equivalent — no errors about missing URL).

- [ ] **Step 4: Verify the app still starts**

Run:
```bash
bun run dev
```
Wait until you see `Ready in <ms>`. Then ⌃C to stop. The app should boot without errors.

- [ ] **Step 5: Commit**

```bash
git add prisma.config.ts .env.example
git commit -m "feat(prisma): support DIRECT_DATABASE_URL for migrations

Required for Neon production where the pooled URL (PgBouncer transaction mode)
doesn't support the prepared statements prisma migrate issues. Locally,
falls back to DATABASE_URL for backwards compatibility."
```

## Task 4: Add Vercel build hooks (postinstall + vercel-build)

**Files:**
- Modify: `package.json`

**Why:** Vercel needs to (1) regenerate the Prisma client after `npm install` (Vercel caches `node_modules` and may skip prisma generate otherwise), and (2) apply pending migrations as part of every deploy. Both happen via standard npm script hooks. Vercel auto-detects `vercel-build` and prefers it over `build`.

- [ ] **Step 1: Modify `package.json` scripts**

Add `postinstall` and `vercel-build` to the `scripts` block. The full updated `scripts` block should read:

```json
"scripts": {
  "dev": "docker compose -f docker/docker-compose.yml up -d && next dev",
  "build": "next build",
  "vercel-build": "prisma migrate deploy && next build",
  "start": "next start",
  "lint": "eslint",
  "test": "vitest run",
  "test:watch": "vitest",
  "test:integration": "vitest run --config vitest.integration.config.ts",
  "postinstall": "prisma generate"
},
```

(Order matters only for human readability — keep the existing entries first.)

- [ ] **Step 2: Verify postinstall doesn't break local installs**

Run:
```bash
bun install
```
Expected: install completes. The `postinstall` step runs `prisma generate` and prints `✔ Generated Prisma Client (v7.x.x) to ./src/generated/prisma`.

- [ ] **Step 3: Verify the production build pipeline works against your local DB**

This simulates what Vercel will run. With your local Postgres still up:

```bash
NODE_ENV=production bun run vercel-build
```

Expected: `prisma migrate deploy` reports `No pending migrations to apply.` (or applies them), then `next build` completes successfully.

- [ ] **Step 4: Commit**

## Rollback Plan (if you need to undo)

If anything goes catastrophically wrong:

- **Bad deploy:** In Vercel **Deployments**, find the last working deployment, click **⋯ → Promote to Production**.
- **Bad migration:** Migrations are forward-only. Roll back via a new migration that reverses the change. For an emergency: `psql "<DIRECT_URL>"` and execute the inverse SQL manually.
- **Compromised credentials:** Rotate the affected secret. For `NEXTAUTH_SECRET` it invalidates all existing sessions (you'll need to sign in again). For `CRON_SECRET` no user impact. For Google OAuth, regenerate the secret in Google Console and update Vercel.
- **Want to fully tear down:** Delete the Vercel project (deployments stop instantly) and the Neon project (data gone). The GitHub repo is untouched.

## What this plan does NOT do (intentionally)

These belong in follow-up plans. Don't scope-creep this one.

1. **Backups** — Polaris stores irreplaceable journal entries. Currently only protected by Neon's 6h branching window. **Plan B**: add a GitHub Actions workflow that runs nightly `pg_dump` → upload to S3 with 30-day retention.
2. **Object storage** — `STORAGE_DRIVER=local` on Vercel writes to `/tmp` which evaporates per-invocation. No system uses uploads today, so this is fine for now. **Plan B (or C)**: provision an S3 bucket in `ap-southeast-1`, IAM user, set `STORAGE_DRIVER=s3` and the S3 env vars in Vercel.
3. **Preview branch DBs per PR** — your workflow currently is "commit to main." Once you start opening PRs (e.g. for risky migrations), each preview deploy should get an isolated Neon branch. **Plan C**: install the Neon Vercel integration in branch-per-deploy mode + set up branch protection on `main` + adopt PR workflow.
4. **Custom domain** — `*.vercel.app` is fine to start. Custom domain can be added in Vercel **Settings → Domains** at any time without redeploying.
5. **Observability** — no error monitoring (Sentry etc.) yet. Add when you actually feel the absence.
